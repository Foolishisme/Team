## 1. 认知升级：为什么不用网页版 GPT？

在网页端，你需要手动复制粘贴代码，这会产生巨大的**信息差**和**执行差**。

- **Context (上下文)**：Cursor 会索引你的整个本地文件夹。它知道你的函数在哪定义，知道你的库是怎么调用的。
    
- **闭环执行**：它不仅能“写”代码，还能直接“改”文件、运行终端命令。
    

## 2. 工程底座：索引与规则 (Indexing & Rules)

为了让 AI 聪明地干活，你必须先给它划定“视线范围”。

### 2.1 `.cursorignore` (信息降噪)

**作用**：告诉 Cursor 哪些文件**不需要**看。

- **必要性**：如果你的项目有巨大的数据文件夹（如 `node_modules` 或 `venv`），不忽略它们会浪费 Token 且干扰 AI 的判断（噪声太大）。
    
- **最佳实践**：在根目录创建 `.cursorignore`，内容通常可以复制 `.gitignore`。
    

### 2.2 `.gitignore` (生存优先)

**作用**：防止敏感信息（API Keys、个人配置）被推送到 GitHub。

- **非对称风险**：一旦 API Key 泄露到公开仓库，你的账户可能在几分钟内被刷爆。
    

### 2.3 `.cursorrules` (系统指令)

**高级技巧**：在根目录创建该文件，写下你希望 Cursor 遵循的风格（如：“始终使用 TypeScript”、“所有变量名必须用驼峰式”）。

## 3. 核心交互：三大快捷键

### 3.1 `Ctrl + K` (行内编辑/生成)

- **场景**：你选中一段代码，想重构它；或者在空行想生成一段逻辑。
    
- **认知点**：它是**局部**的。它只关注你当前选中的内容。
    

### 3.2 `Ctrl + L` (侧边栏对话 - Chat)

- **场景**：问问题。“这个函数是什么意思？”、“怎么实现登录逻辑？”。
    
- **核心功能 - `@` 符号**：
    
    - `@Files`：指定让它看某个文件。
        
    - `@Codebase`：让它搜索全库。**这是最强大的功能**，能消除你和项目之间的信息差。
        
    - `@Docs`：引用外部文档（如 React 官网）。
        

### 3.3 `Ctrl + I` (Composer - 核心利器)

- **场景**：**跨文件修改**。你想做一个“增加用户注册功能”，涉及到前端、后端、数据库。
    
- **操作**：直接描述需求，它会同时修改多个文件。
    

## 4. 模式解析：四种智能模式

在 `Ctrl + I` (Composer) 或 `Ctrl + L` 中，你可以切换模式。

### 4.1 Normal / Ask (问答模式)

- 只说不做。适合初学者理解代码逻辑。
    

### 4.2 Agent (执行模式 - 推荐)

- **认知**：AI 拥有了“手”。
    
- **行为**：它会自己读取文件、运行终端测试、根据报错信息自动修复代码。
    
- **建议**：当你有一个明确的开发任务（如“修复这个 Bug”）时，直接用 Agent 模式。
    

### 4.3 Plan (规划模式)

- **逻辑**：在动手前先思考。
    
- **适用**：复杂功能开发。它会先列出步骤（修改哪些文件、逻辑怎么走），得到你的认可后再执行。
    
- **非对称优势**：这避免了 AI 盲目改写代码导致的系统性崩溃。
    

### 4.4 Bug (排查模式)

- 专门用于定位错误。它会扫描相关文件，寻找逻辑漏洞。
    

## 5. 包库用户规则 (Docs & Libraries)

如果你在使用一些最新的库（AI 训练数据可能滞后），请使用 **Docs 功能**：

1. 按 `Ctrl + L`。
    
2. 输入 `@Docs` -> `Add new doc`。
    
3. 粘贴该库的官方文档 URL。
    
4. **效果**：Cursor 会即时爬取文档，确保它写出的代码是 **State of the Art**，而不是过时的旧语法。
    

## 6. 系统执行 SOP (建议同事遵循)

1. **先 Pull 代码**：确保你在最新版本上操作。
    
2. **开启 Composer (Ctrl + I)**：切换到 **Agent** 模式。
    
3. **清晰描述**：描述你的目标（如“在首页添加一个搜索框，调用后端 /search 接口”）。
    
4. **Review 改动**：Cursor 改完后，会显示 Diff（红绿对比）。**一定要看一眼**，确认逻辑没问题再点击 `Accept`。
    
5. **运行测试**：让 Agent 在终端运行测试，确保没有破坏现有系统。
    

> 💡 聪生活提示：
> 
> Cursor 是你的“外挂大脑”。与其花 2 小时查文档改 Bug，不如花 2 分钟写好 Prompt 让 Agent 去跑。省下的时间去进行更高维度的系统思考。
